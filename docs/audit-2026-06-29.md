## 🔴 CRITICAL — Fix Before Any Deploy

### C-1 · `product_images` table has no migration  
**Risk:** Every product page, homepage, RFQ detail, and admin product view will 500 or return no images.  
**Evidence:** `product_images` is referenced in **34+ places** across `src/` (products route, product detail page, admin images API, HomeSections, RFQ route, quotations, etc.) but there is **no CREATE TABLE for `product_images`** anywhere in `homeu-schema.sql` or any of the 50 migration files (`001`–`049`). It is not in the initial schema dump.  
**Fix:** Write and apply a new migration (e.g., `050_product_images_table.sql`):
```sql
CREATE TABLE IF NOT EXISTS product_images (
  id SERIAL PRIMARY KEY,
  product_id INTEGER NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  url TEXT NOT NULL,
  alt TEXT DEFAULT '',
  sort_order INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS product_images_product_id_idx ON product_images(product_id);
```
Apply to **both local and production** before next deploy.

### C-2 · Duplicate migration files with same number  
**Risk:** Migration runner will execute one and skip or error on the other; one backfill will never run.  
**Evidence:** Both `049_backfill_product_ratings.sql` and `049_backfill_review_product_ids.sql` have the prefix `049`. The migrate runner (`tools/migrate/migrate.mjs`) almost certainly keys on filename; one of these will be silently ignored or cause a conflict.  
**Fix:** Rename one to `050_backfill_review_product_ids.sql` (after resolving C-1, bump to `051`). Audit migrate.mjs to confirm how it tracks applied migrations (by name or by number).

### C-3 · `homeu-schema.sql` is a stale PayloadCMS snapshot — not the live schema  
**Risk:** Anyone following the README quickstart (`psql $DATABASE_URI < homeu-schema.sql`) will get an incomplete, wrong database. Production has 50 migrations on top of this file; the snapshot is from an early PayloadCMS era.  
**Evidence:** Schema contains `DaVinciOS_kv`, `DaVinciOS_locked_documents`, `DaVinciOS_migrations`, `DaVinciOS_preferences` (PayloadCMS internal tables). These are dead framework tables. It does NOT contain `customers`, `quotations`, `redirects`, `newsletter_subscribers`, `product_images`, `rfq_request_items`, or any of the ~40 tables added by migrations 002–049.  
**Fix:**
1. Run a fresh `pg_dump --schema-only` from **production** after all migrations are applied.
2. Replace `homeu-schema.sql` with that dump.
3. Update the README to say: "Apply schema via migrations: `node tools/migrate/migrate.mjs`" instead of pointing at the SQL file directly.
4. Consider creating a `tools/migrate/migrations/000_baseline.sql` that is the full idempotent schema for a fresh install.

### C-4 · `customers` table missing from `homeu-schema.sql` initial migration  
**Risk:** The `001_initial_schema.sql` does not define a `customers` table. Every migration from `003` onward references `customers`. A fresh install from `001` alone would fail on `003+`.  
**Evidence:** Grep of `homeu-schema.sql` shows only: `categories`, `media`, `pages`, `products`, `products_rels`, `rfq_requests`, `rfq_requests_items`, `users`, `users_sessions`, `DaVinciOS_*`. No `customers` table.  
**Fix:** Either add `customers` to the `000`/`001` baseline, or document explicitly that migrations must be run in sequence from 001–049 and a fresh DB will apply all of them in order. Verify that `migrate.mjs` handles a truly empty DB.

## 🟠 HIGH — Likely to Break Live Flows

### H-1 · `auth.ts` queries the `customers` table for admin login  
**Risk:** Admin login (`authenticateAdmin`) SELECTs from `customers` WHERE `status = 'active'`. Admin users must have a `role` in `('admin','superadmin','editor','sales')` AND a `password_hash` column AND a `tab_permissions` column. If any of these columns are missing (they are added by later migrations, not the base schema), login will silently fail with "Invalid email or password."  
**Evidence:** `auth.ts` lines 105–111: `SELECT id, email, name, role, password_hash, tab_permissions FROM customers WHERE email = $1 AND status = 'active'`.  
**Fix:** Verify migrations 009, 020+ have been applied to production. Run `\d customers` on the production DB and confirm all these columns exist: `role`, `password_hash`, `tab_permissions`, `status`, `name`, `company`.

---

### H-2 · `chatbot` schema may not be initialized on production  
**Risk:** The chatbot schema (`chatbot.leads`, `chatbot.conversations`, `chatbot.messages`, `chatbot.appointments`, etc.) is defined in `src/lib/chatbot/schema.sql` as a separate SQL file — not wired into the numbered migration sequence. It lives outside `tools/migrate/migrations/`. If it was never manually applied, every chatbot API route, the entire leads/analytics section of admin, the RFQ route (which LEFT JOINs `chatbot.leads`), and appointment scheduling will error.  
**Evidence:** `src/lib/chatbot/schema.sql` defines 10+ tables under the `chatbot` schema. It is not referenced by `migrate.mjs`. `036_chatbot_appointments.sql` adds a column to `chatbot.appointments` which presupposes the schema already exists.  
**Fix:** Add a migration `050_chatbot_schema_baseline.sql` (or numbered appropriately) that runs the chatbot schema SQL in idempotent form. Ensure `migrate.mjs` runs it before `036`.

---

### H-3 · `rfq_request_items` vs `rfq_requests_items` naming conflict in the schema  
**Risk:** The base schema (`homeu-schema.sql` / `001_initial_schema.sql`) defines `rfq_requests_items` (PayloadCMS underscore style). The application code and migrations 003/024 use `rfq_request_items` (without the extra `s`). If the old PayloadCMS table still exists and the new one was never created, every RFQ detail fetch and item insert will fail.  
**Evidence:** Comment in `003_add_rfq_columns.sql`: *"The codebase references `rfq_request_items` but only `rfq_requests_items` (PayloadCMS naming) exists in the schema."*  
**Fix:** Confirm `rfq_request_items` exists in production (`\dt rfq_request*`). Migration 003 should have created it. Verify it was applied. If `rfq_requests_items` also still exists, verify it can be dropped (check if any code still uses the old name).

---

### H-4 · SMTP config falls back silently — emails may never arrive  
**Risk:** `smtp-config.ts` loads SMTP from the `DaVinciOS_kv` table (DB-stored config), falling back to env vars. If neither is configured, `smtp.user` and `smtp.pass` are empty strings and the transporter will fail silently (caught and logged, not thrown). Customer confirmation emails and admin RFQ notification emails will be silently dropped.  
**Evidence:** `smtp-config.ts` lines 49–50: falls back to `process.env.SMTP_USER || process.env.SALES_EMAIL || ''`. `.env` shows `SALES_EMAIL_PASS=your-zoho-app-password` (placeholder). The `DaVinciOS_kv` key `smtp_config` may not be set on a fresh install.  
**Fix:**
1. Set real SMTP credentials via the admin Settings panel (which writes to `DaVinciOS_kv`).
2. Add `SMTP_HOST`, `SMTP_USER`, `SMTP_PASS` to `.env` as production fallbacks.
3. Add an `/api/admin/settings/test-email` endpoint if one doesn't exist, so the owner can verify email delivery from the admin UI without writing a test script.
4. Log a clear warning at startup if both DB and env SMTP credentials are empty.

---

### H-5 · `middleware.ts` throws at module load if `JWT_SECRET` is missing  
**Risk:** Line 9 of `middleware.ts` throws `new Error('JWT_SECRET is not configured...')` at module-level (not inside the function). In Next.js App Router, middleware is loaded for every request. If `JWT_SECRET` is undefined (e.g., a misconfigured deploy), every single request to the app will 500 with an unhelpful error, including the health check endpoint.  
**Note:** `auth.ts` correctly defers this to a `getJwtSecret()` function. Middleware does not.  
**Fix:** Move the JWT_SECRET validation inside the `middleware()` function body (or use the same deferred pattern from `auth.ts`). The error should still throw, but not at module-import time.

---

### H-6 · `rfq_request_items` lacks `accepts_alternatives` column in official migration  
**Risk:** The route handler at `api/rfq/route.ts` (line 241) inserts `accepts_alternatives BOOLEAN NOT NULL DEFAULT true` — this column exists in the inline CREATE TABLE fallback, but not in migration `003`. If the table was created by migration 003 (no `accepts_alternatives`) and later requests try to INSERT with that column, they will 500.  
**Fix:** Add to a new migration: `ALTER TABLE rfq_request_items ADD COLUMN IF NOT EXISTS accepts_alternatives BOOLEAN NOT NULL DEFAULT true;`

### M-9 · `bcryptjs.hashSync` called in an async context  
**Evidence:** `auth.ts` line 29: `return bcrypt.hashSync(password, 10)` inside an `async function`. `hashSync` is a CPU-blocking call. In Node.js, this blocks the event loop during password hashing (which is intentionally slow at cost factor 10).  
**Fix:** Use `bcrypt.hash(password, 10)` (async version) and `await` it.

**Fix:** Read `migrate.mjs` and verify: (a) it records applied migrations by filename, (b) re-running is idempotent, (c) all 50 migrations are tracked.

### I-3 · No health check defined in Dockerfile or `docker-compose.yml`  
**Risk:** Docker will restart the container on crash, but has no way to know if the app is actually serving requests (it could be stuck in a startup loop).  
**Fix:** Add to `docker-compose.yml`:
```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:3000/api/health"]
  interval: 30s
  timeout: 10s
  retries: 3
  start_period: 60s
```
The `/api/health` route already exists in the codebase.

---

### I-4 · `postgres` service has no health check  
**Risk:** The `website` service starts with `depends_on: - postgres`, but Docker `depends_on` only waits for the container to start, not for PostgreSQL to be ready to accept connections.  
**Fix:** Add a healthcheck to the postgres service:
```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U homeu -d homeu"]
  interval: 5s
  timeout: 5s
  retries: 10
```
And change `website`'s `depends_on` to use `condition: service_healthy`.

---

### I-5 · `ollama` service starts unconditionally, consuming memory  
**Evidence:** `docker-compose.yml` always starts the Ollama service. In production where `AI_PROVIDER=ollama` might not be the chosen provider (e.g., Gemini is used), this wastes RAM.  
**Fix:** Add a Docker profile so Ollama only starts when explicitly requested: `profiles: ["ollama"]`. Or add documentation noting that if Gemini/OpenAI is used, Ollama can be removed from compose.

### S-2 · Session cookie missing `__Secure-` prefix enforcement  
**Evidence:** Cookie name is `homeu_admin_session`. In `auth.ts`, `secure: process.env.NODE_ENV === 'production'` is set, which is correct. However, the cookie name is not prefixed with `__Secure-` or `__Host-`, which would enforce browser-level security requirements.  
**Fix (optional hardening):** Rename cookie to `__Host-homeu_admin` to trigger browser strict security. Not critical but adds defense in depth.

---

### S-3 · No CSRF protection on state-mutating API routes  
**Risk:** Admin API routes (POST/PUT/DELETE to `/api/admin/*`) rely on JWT cookies but have no CSRF token check. A malicious site could trigger admin actions if the admin is logged in.  
**Fix:** Add `SameSite: Strict` to the session cookie (currently `lax`), or add CSRF token verification on admin mutation routes. `SameSite: Strict` would break OAuth redirect flows — instead verify `Origin`/`Referer` headers on mutation routes.

---

### S-4 · SQL injection risk in `lib/db.ts` `find()` `opts.orderBy`  
**Evidence:** `db.ts` line 73: `if (opts.orderBy) sql += \` ORDER BY ${opts.orderBy}\`` — this interpolates the `orderBy` value directly into SQL without quoting. If any caller passes user-controlled input as `orderBy`, it is injectable.  
**Fix:** Either whitelist allowed orderBy column names, or use a column-quoting helper. Audit all callers of `find()` to confirm none pass user input as `orderBy`.
